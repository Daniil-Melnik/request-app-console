<h1>Система автоматизации регулирования и контроля выполнения заявок в организации</h1>
<h2>Краткое описание возможностей и требований из ТЗ</h2>
<p>Изложенный в репозитории проект представляет реализацию простой системы управления заявками в виде приложения с консольным вводом/выводом. Программа отвечает ряду требований изложенных далее.</p>
<p>
<ul>
  <li>Заявка проходит следующие этапы: Новая → В работе → Выполнена. Переход между статусами должен соответствовать бизнес-правилам. Например, заявка не может быть переведена из статуса «Новая» сразу в статус «Выполнена».</li>
  <li>Присутсвует справочник сотрудников, в котором указаны ФИО, должность и подразделение</li>
  <li>Присутсвует каталог созданных заявок. У каждой заявки есть номер, дата создания, автор, исполнитель, описание, срок выполнения и статус</li>
  <li>При работе с заявками можно: корректно переводить заявку из одного статуса в другой, переназначение исполнителя, вывести список заявок по фильтру (статус, исполнитель, подразделение, просроченные заявки)</li>
  <li>Присутвует возможность формирования отчёта, содержащего: количество заявок по каждому статусу; количество просроченных заявок; количество выполненных заявок по исполнителям</li>
</ul>
</p>
<h2>ПО применённое при разработке</h2>
<p>В ходе реализации системы применено следующее программное обеспечение: 
  <ul>
    <li>ОС Windows 10</li>
    <li>IDE Intellij idea 2025.2 CE</li>
    <li>СУБД PostgreSQL 17.10</li>
    <li>OpenJDK 11</li>
  </ul>
</p>
<h2>Описание архитектуры и реализации системы</h2>
<p>В состав программного комплекса входят две части: база данных и программа-консольная утилита. Программа в свою очередь включает в себя организацию сущностей "Запрос (Request)", "Сотрудник (Employee)" и "Экземпляр истории изменения статусов (StatusHistory)"
<p></p>
<p>Указанные сущности описаны в каталоге <a href="src/main/java/medan/model">src/main/java/medan/model</a>. При реализации взаимодействия был применён механиз Hibernate ORM. В кажом классе можно найти описание каркаса БД в виде аннотаций. Базовые дампы можно найти в <a href="request_db_init_with_index.sql">request_db_init_with_index.sql</a> и <a href="request_db_init_without_index.sql">request_db_init_without_index.sql</a></p>.
<p>Взаимодействие с СУБД организовано средствами двух прослоек: базовой через Entity-Manager + DAO (<a href="src/main/java/medan/dao">src/main/java/medan/dao</a>, <a href="src/main/java/medan/util/JpaUtil.java">src/main/java/medan/util/JpaUtil.java</a>) и более абстрактной - Service (<a href="src/main/java/medan/service">src/main/java/medan/service</a>)</p>
<p></p>При использовании программы можно выполнять следующие действия:
<u>
  <li>Добавить сотрудника</li>
  <li>Редактировать информацию о сотруднике</li>
  <li>Создать заявку</li>
  <li>Изменить статус заявки</li>
  <li>Назначить исполнителя заявки</li>
  <li>Показать несколько заявок с возможностью фильтрации</li>
  <li>Посмотреть одну заявку по номеру</li>
  <li>Показать список сотрудников</li>
  <li>Вывести на экран отчёт</li>
  <li>Выполнить заполнение БД тестовыми данными</li>
  <li>Замерить производительность</li>
  <li>Просмотреть историю измененения статусов всех заявок или конкретной по номеру</li>
</u>
</p>

<h2>Описание модели </h2>
=====================================================================================
Требования к базе данных

Спроектировать структуру базы данных таким образом, чтобы:
отсутствовало необоснованное дублирование данных;
использовались связи между сущностями;
соблюдались принципы нормализации.

Необходимо кратко объяснить принятые решения по структуре базы данных.
Производительность

Заполнить базу тестовыми данными:
не менее 1 000 000 заявок;
не менее 1000 сотрудников.

Выполнить запрос:
вывести все просроченные заявки конкретного исполнителя, находящиеся в статусе «В работе», отсортированные по сроку выполнения.

Замерить время выполнения запроса.
После этого выполнить оптимизацию предложив и описав методы оптимизации, повторить замер и кратко описать:
какие изменения были внесены;
почему они ускоряют выполнение;
результаты измерений до и после оптимизации.

Технические требования
Использовать любой объектно-ориентированный язык программирования.
Использовать SQL СУБД или MongoDB.
Обязательно реализовать объектную модель предметной области.
Код должен быть структурирован и легко расширяем.
Результат выполнения

Предоставить:
исходный код;
инструкцию по запуску;
краткое описание реализованного бизнес-процесса;
описание реализованных бизнес-правил;
описание выполненной оптимизации базы данных и результатов замеров производительности.

Дополнительное задание (по желанию)

Опишите в письме, какие изменения потребуется внести в систему, если:

заявка должна проходить согласование руководителем;
у заявки может быть несколько исполнителей;
необходимо хранить историю изменения статусов;
сроки выполнения зависят от типа заявки;
требуется разграничение прав доступа (сотрудник, руководитель, администратор).
