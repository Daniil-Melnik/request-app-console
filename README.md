<h2>Краткое описание возможностей и требований из ТЗ</h2>
<p>Изложенный в репозитории проект представляет реализацию простой системы управления заявками в виде приложения с консольным вводом/выводом. Программа отвечает ряду требований изложенных далее.</p>
<p>
<ul>
  <li>Заявка проходит следующие этапы: Новая → В работе → Выполнена. Переход между статусами должен соответствовать бизнес-правилам. Например, заявка не может быть переведена из статуса «Новая» сразу в статус «Выполнена».</li>
  <li>Присутсвует справочник сотрудников, в котором указаны ФИО, должность и подразделение</li>
  <li>Присутсвует каталог созданных заявок. У каждой заявки есть номер, дата создания, автор, исполнитель, описание, срок выполнения и статус</li>
  <li>При работе с заявками можно: корректно переводить заявку из одного статуса в другой, переназначение исполнителя, вывести список заявок по фильтру (статус, исполнитель, подразделение, просроченные заявки)</li>
  <li>Присутвует возможность формирования отчёта, содержащего: количество заявок по каждому статусу; количество просроченных заявок; количество выполненных заявок по исполнителям</li>
</ul>
</p>

<h2>Описание архитектуры и реализации системы</h2>
<p>В состав программного комплекса входят две части: база данных и программа-консольная утилита. Программа в свою очередь включает в себя организацию сущностей "Запрос (Request)", "Сотрудник (Employee)" и "Экземпляр истории изменения статусов (StatusHistory)"
<p></p>
<p>Указанные сущности описаны в каталоге <a href="src/main/java/medan/model">src/main/java/medan/model</a>. При реализации взаимодействия был применён механизм Hibernate ORM. В кажом классе можно найти описание каркаса БД в виде аннотаций. Базовые дампы можно найти в <a href="request_db_init_with_index.sql">request_db_init_with_index.sql</a> и <a href="request_db_init_without_index.sql">request_db_init_without_index.sql</a></p>.
<p>Взаимодействие с СУБД организовано средствами двух прослоек: базовой через Entity-Manager + DAO (<a href="src/main/java/medan/dao">src/main/java/medan/dao</a>, <a href="src/main/java/medan/util/JpaUtil.java">src/main/java/medan/util/JpaUtil.java</a>) и более абстрактной - Service (<a href="src/main/java/medan/service">src/main/java/medan/service</a>)</p>
<p>На старте и при каждой итерации пользователю выводится меню, пункты которого разобраны по категория. Работа с системой осуществляется через ввод номера пункта меню, соответсвующего нужному действию.Программа обеспечивает бесперебойную работу при некорректном вводе. Местами возможна необходимость дополнительного нажания на клавишу "Enter"</p>

<h2>Развёртывание системы</h2>
<p>Развёртывание происходит в два этапа: настройка СУБД PostgreSQL на локальной машине и запуск консольной утилиты</p>
<p></p>
<p>Развёртывание БД выполняется следующим образом: </p>
<ul>
  <li><a href="https://www.postgresql.org/download/">Скачать</a> и установить СУБД PostgreSQL на ПЭВМ</li>
  <li>Создать базу данных request_db</li>
  <li>Восстановить любой бэкап из корня репозитория, например чистый бэкап: 
    <ul>
      <li><code>cd path_to_repo/request-app-console</code></li>
      <li><code>psql -U postgres -p 5433 -d request_db < request_db_init_without_index.sql</code></li>
    </ul>
  </li>
  <li>БД должна соответсвовать требованиям:
  <ul>
    <li>IP-адрес сервера: 127.0.0.1</li>
    <li>Порт сервера: 5433</li>
    <li>Имя БД: request_db</li>
    <li>Пользователь БД: postgres</li>
    <li>Пароль доступа БД: 12345678</li>
  </ul>
  </li>
</ul>
<p>Запуск программы можно осуществить либо через запуск исполняемого jar-файла: <code>java -jar request-app-console.jar</code></p>
<p>Также запуск можно осуществить из IDE inteliJ idea</p>

<h2> Идеи усовершенствования структуры БД</h2>
<p>В данном проекте рассматривается два случая организации базы данных: "в лоб", когда данные просто хранятся в таблицах и оптимизированные - когда помимо таблиц в БД присутсвую индексы по этим таблицам.</p>
<p>Для быстрого восстановлени и воспроизводимости БД с индексами и без в корне репозитория лежат чистые .sql-дампы баз данных для обоих случаем</p>
<p>Для оптимизации работы по <code>WHERE</code> и <code>ORDER BY</code> были созданы индексы :</p>
<p>Основной: </p>
<ul>
  <li><code>CREATE INDEX idx_requests_executor_status_due ON requests (executor_id, status, due_date)</code></li>
</ul>
<p>Дополнительные: </p>
<ul>
  <li><code>CREATE INDEX idx_requests_executor_status ON requests (executor_id, status)</code></li>
  <li><code>CREATE INDEX idx_requests_due_date ON requests (due_date)</code></li>
  <li><code>CREATE INDEX idx_requests_status ON requests (status)</code></li>
</ul>
<h2>Замеры производительности</h2>
<p>Замер выполняется следующим образом: создаётся пустая БД request_db -> наполняется средствами системы нужным числом записей -> средствами системы выполняется тестовый запрос</p>
<p>Ниже приведены результаты вычисления производительности на 1000 сотрудников и 1_000_000 записей. Сначала без индексов, потом с индексами</p>
<img width="312" height="426" alt="without_indexes" src="https://github.com/user-attachments/assets/13f9249b-2421-4308-ac43-502864ef3256" />
<img width="317" height="427" alt="with index" src="https://github.com/user-attachments/assets/0b4086e4-d05a-478f-8a92-bf936474e40e" />
<p>По результатам видно, что без индексов время всех запросов примерно одно и тоже. В тоже время с индексом самый долгий запрос - первый, остальные - проходят в 100 раз быстрее чем было без индекса. Мы наблюдаем классический лабораторный случай ускорения работы системы после оптимизации базы данных.</p>
